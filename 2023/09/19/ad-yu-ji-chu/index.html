1.介绍

Active Directory （AD） 是用于 Windows 网络环境的目录服务。它是一种分布式分层结构，允许集中管理组织的资源，包括用户、计算机、组、网络设备、文件共享、组策略、设备和信任。AD 在 Windows 域环境中提供身份验证和授权功能。近年来，它受到越来越多的攻击。它被设计为向后兼容，许多功能可以说不是“默认安全的”，并且很容易配置错误。可以利用此弱点在网络中横向和垂直移动并获得未经授权的访问。AD 本质上是一个相当大的只读数据库，可供域中的所有用户访问，无论其权限级别如何。没有添加权限的基本 AD 用户帐户可以枚举 AD 中的大多数对象。这一事实使得正确保护 AD 实现变得极其重要，因为任何用户帐户，无论其权限级别如何，都可用于枚举域并彻底查找错误配置和缺陷。此外，仅使用标准域用户帐户即可执行多次攻击，这表明纵深防御策略和仔细规划的重要性，重点是安全性和强化 AD、网络分段和最小特权。

2.AD基础知识

2.1 AD结构

Active Directory Domain Services Overview | Microsoft Learn

Active Directory （AD） 是用于 Windows 网络环境的目录服务。它是一种分布式分层结构，允许集中管理组织的资源，包括用户、计算机、组、网络设备和文件共享、组策略、服务器和工作站以及信任。AD 在 Windows 域环境中提供身份验证和授权功能。目录服务（如 Active Directory 域服务 （AD DS））为组织提供了存储目录数据的方法，并使其可供同一网络上的标准用户和管理员使用。AD DS 存储用户名和密码等信息，并管理授权用户访问此信息所需的权限。它最初随Windows Server 2000一起提供;近年来，它受到越来越多的攻击。它被设计为向后兼容，许多功能可以说不是“默认安全的”。很难正确管理，尤其是在容易错误配置的大型环境中。
Active Directory 缺陷和错误配置通常可用于获得立足点（内部访问），在网络中横向和垂直移动，以及未经授权访问受保护的资源，如数据库、文件共享、源代码等。AD 本质上是一个大型数据库，可供域中的所有用户访问，无论其权限级别如何。没有添加权限的基本 AD 用户帐户可用于枚举 AD 中包含的大多数对象，包括但不限于：

Domain Computers 域计算机 Domain Users 域用户
Domain Group Information 域组信息 Organizational Units (OUs) 组织单位 （OU）
Default Domain Policy 默认域策略 Functional Domain Levels 功能域级别
Password Policy 密码策略 Group Policy Objects (GPOs) 组策略对象 （GPO）
Domain Trusts 域名信任 Access Control Lists (ACLs) 访问控制列表 （ACL）

Active Directory 以分层树结构排列，顶部的林包含一个或多个域，这些域本身可以具有嵌套的子域。林是所有对象都处于管理控制之下的安全边界。一个林可能包含多个域，一个域可能包含其他子域或子域。域是可在其中访问包含的对象（用户、计算机和组）的结构。它具有许多内置的组织单位 （OU），例如 Domain Controllers 、 Users ， Computers 并且可以根据需要创建新的 OU。OU 可能包含对象和子 OU，允许分配不同的组策略。

一个非常简单的AD域结构如下：

INLANEFREIGHT.LOCAL/
├── ADMIN.INLANEFREIGHT.LOCAL
│   ├── GPOs
│   └── OU
│       └── EMPLOYEES
│           ├── COMPUTERS
│           │   └── FILE01
│           ├── GROUPS
│           │   └── HQ Staff
│           └── USERS
│               └── barbara.jones
├── CORP.INLANEFREIGHT.LOCAL
└── DEV.INLANEFREIGHT.LOCAL

在这里，我们可以说这是 INLANEFREIGHT.LOCAL 根域，包含子域（子域或树根域） ADMIN.INLANEFREIGHT.LOCAL ，以及 DEV.INLANEFREIGHT.LOCAL 构成域的其他对象，例如用户，组，计算机等， CORP.INLANEFREIGHT.LOCAL 我们将在下面详细介绍。在大型组织中，通常会看到多个域（或林）通过信任关系链接在一起。与在当前域中重新创建所有新用户相比，与另一个域/林创建信任关系通常更快、更容易。如果管理不当，域信任可能会引入一系列安全问题。

下面的图表展示了两个林（forests）：INLANEFREIGHT.LOCAL 和 FREIGHTLOGISTICS.LOCAL。双向箭头表示这两个林之间存在双向信任，这意味着在INLANEFREIGHT.LOCAL的用户可以访问FREIGHTLOGISTICS.LOCAL中的资源，反之亦然。我们还可以看到在每个根域下有多个子域。在这个示例中，我们可以看到根域信任每个子域，但林A中的子域不一定已经与林B中的子域建立了信任关系。这意味着，即使在顶级inlanefreight.local和freightlogistics.local域之间存在双向信任，但属于admin.dev.freightlogistics.local的用户默认情况下无法验证到wh.corp.inlanefreight.local域中的计算机。要允许admin.dev.freightlogistics.local和wh.corp.inlanefreight.local之间的直接通信，需要建立另一个信任关系。

2.2 AD术语

2.2.1 Object(对象)

对象可以定义为 Active Directory 环境中存在的任何资源，例如 OU、打印机、用户、域控制器等。

2.2.2 Attributes(属性)

All Attributes - Win32 apps | Microsoft Learn

Active Directory 中的每个对象都有一组关联的属性，用于定义给定对象的特征。计算机对象包含主机名和 DNS 名称等属性。AD 中的所有属性都有一个关联的 LDAP 名称，可以在执行 LDAP 查询时使用，例如 displayName 关联Full Name ，given name 关联First Name 。

2.2.3 Schema(图式)

Schema (AD DS) - Win32 apps | Microsoft Learn

Active Directory 架构本质上是任何企业环境的蓝图。它定义 AD 数据库中可以存在的对象类型及其关联属性。它列出了与 AD 对象对应的定义，并保存有关每个对象的信息。例如，AD 中的用户属于类“用户”，计算机对象属于“计算机”类，依此类推。每个对象都有自己的信息（有些需要设置，有些是可选的），这些信息存储在属性中。当从类创建对象时，这称为实例化，从特定类创建的对象称为该类的实例。例如，如果我们使用计算机RDS01。此计算机对象是 Active Directory 中“计算机”类的实例。

2.2.4 Domain(域)

域是对象的逻辑组，例如计算机、用户、OU、组等。我们可以将每个域视为州或国家/地区的不同城市。域可以完全相互独立运行，也可以通过信任关系进行连接。

2.2.5 Forest(森林)

林是 Active Directory 域的集合。它是最顶层的容器，包含下面介绍的所有 AD 对象，包括但不限于域、用户、组、计算机和组策略对象。林可以包含一个或多个域，并被视为美国的一个州或欧盟内的国家/地区。每个林独立运行，但可能与其他林具有各种信任关系。

2.2.6 Tree(树)

树是从单个根域开始的AD域的集合。森林是 AD 树的集合。树中的每个域与其他域共享一个边界。当一个域添加到树中的另一个域下时，将形成父信任关系。同一林中的两棵树不能共享名称（命名空间）。假设我们在 AD 林中有两棵树： inlanefreight.local 和 ilfreight.local 。第一个的子域可以是 corp.ilfreight.local ， 而第二个的子域可以是corp.inlanefreight.local 。树中的所有域共享一个标准全局编录（AD Global Catalog），其中包含有关属于该树的对象的所有信息。

2.2.7 Container(容器)

容器对象包含其他对象，并在目录子树层次结构中具有定义的位置。

2.2.8 Leaf(叶)

叶对象不包含其他对象，位于子树层次结构的末尾。

2.2.9 Global Unique Identifier (GUID)(全局唯一标识符(GUID))

Object-Guid attribute - Win32 apps | Microsoft Learn

GUID 是在创建域用户或组时分配的唯一 128 位值。此 GUID 值在整个企业中是唯一的，类似于 MAC 地址。Active Directory 创建的每个对象都分配了一个 GUID，而不仅仅是用户和组对象。GUID 存储在属性ObjectGUID中。查询 AD 对象（如用户、组、计算机、域、域控制器等）时，可以使用 PowerShell 查询其值，也可以通过指定其 objectGUID值、可分辨名称、GUID、SID 或 SAM 帐户名称来搜索它。AD 使用 GUID 在内部标识对象。在 Active Directory 中按 GUID 值搜索可能是查找要查找的确切对象的最准确、最可靠的方法，尤其是在全局编录可能包含对象名称的类似匹配项时。在执行 AD 枚举时指定 ObjectGUID 值将确保我们获得与要搜索信息的对象相关的最准确结果。只要该对象存在于域中，该 ObjectGUID 属性就永不会更改并与该对象关联。

2.2.10 Security principals(安全主体)

Security principals | Microsoft Learn

Security Account Manager - Wikipedia

安全主体是操作系统可以进行身份验证的任何内容，包括用户、计算机帐户，甚至是在用户或计算机帐户上下文中运行的线程/进程（即在域中的服务帐户上下文中运行的应用程序，如 Tomcat）。在 AD 中，安全原则是可以管理对域中其他资源的访问的域对象。我们还可以使用本地用户帐户和安全组来控制对该特定计算机上资源的访问。这些不是由 AD 管理，而是由安全帐户管理器 （SAM） 管理。

2.2.11 Security Identifier (SID)(安全标识符(SID))

Security principals | Microsoft Learn

[LDAPWiki：Well-known Security Identifiers](https://ldapwiki.com/wiki/Wiki.jsp?page=Well-known Security Identifiers)

安全标识符或 SID 用作安全主体或安全组的唯一标识符。每个帐户、组或进程都有自己唯一的 SID，在 AD 环境中，该 SID 由域控制器颁发并存储在安全数据库中。一个 SID 只能使用一次。即使删除了安全原则，也永远无法在该环境中再次使用它来标识其他用户或组。当用户登录时，系统会为他们创建一个访问令牌，其中包含用户的 SID、已授予的权限以及用户所属的任何组的 SID。每当用户在计算机上执行操作时，此令牌用于检查权限。还有一些用于标识通用用户和组的已知 SID。这些在所有操作系统中都是相同的。 一个例子是Everyone组。

2.2.12 Distinguished Name (DN)

Distinguished Names | Microsoft Learn

Distinguished Name描述 AD 中对象的完整路径（例如 cn=bjones, ou=IT, ou=Employees, dc=inlanefreight, dc=local ）。在此示例中，用户 bjones 在 Inlanefreight 公司的 IT 部门工作，他的帐户是在持有公司员工帐户的组织单位 （OU） 中创建的。公用名 （CN） bjones 只是在域中搜索或访问用户对象的一种方法。

2.2.13 Relative Distinguished Name (RDN)

Relative Distinguished Name是Distinguished Name的单个组件，用于将对象标识为与命名层次结构中当前级别的其他对象不同。在我们的示例中， bjones 是对象的RDN。AD 不允许在同一父容器下使用两个具有相同名称的对象，但可以有两个具有相同 RDN 的对象，并且在域中仍然是唯一的，因为它们具有不同的 DN。例如，对象 cn=bjones,dc=dev,dc=inlanefreight,dc=local 与cn=bjones,dc=inlanefreight,dc=local将被视为不同的对象 。

2.2.14 sAMAccountName

User Naming Attributes - Win32 apps | Microsoft Learn

sAMAccountName 是用户的登录名。在这里是 bjones .它必须是唯一值并且小等于20个字符。

2.2.15 userPrincipalName(用户主体名称)

Active Directory: User Principal Name - TechNet Articles - United States (English) - TechNet Wiki (microsoft.com)

属性是在 AD 中标识用户的另一种方法。此属性由前缀（用户帐户名）和后缀（域名）组成，格式为bjones@inlanefreight.local。此属性不是必需的。

2.2.16 FSMO Roles

Active Directory Flexible Single Master Operation (FSMO) roles in Windows - Windows Server | Microsoft Learn

在 AD 的早期，如果一个环境中有多个 DC，它们可能会争夺权限来进行更改，有时更改可能无法正确进行。微软随后实施了”last writer wins”的机制，但如果最后的更改导致问题，这也可能引入自己的问题。然后，他们引入了一个模型，在该模型中，单个“主”DC可以将更改应用于域，而其他DC仅满足身份验证请求。这是一个有缺陷的设计，因为如果主 DC 出现故障，则在恢复之前无法对环境进行任何更改。为了解决这种单点故障模型，Microsoft将 DC 可能具有的各种职责分离为灵活的单主机操作 （FSMO） 角色。这使域控制器 （DC） 能够继续对用户进行身份验证并授予权限而不会中断（授权和身份验证）。有五个FSMO角色：模式主控制器（Schema Master）和域命名主控制器（Domain Naming Master）（每个森林一个），相对标识（RID）主控制器（RID Master）（每个域一个），主域控制器模拟器（PDC Emulator）（每个域一个）和基础结构主控制器（Infrastructure Master）（每个域一个）。所有五个角色都分配给新 AD 林中林根域中的第一个 DC。每次将新域添加到林时，只会将 RID 主机、PDC 仿真程序和基础结构主机角色分配给新域。FSMO 角色通常在创建域控制器时设置，但系统管理员可以根据需要转移这些角色。这些角色有助于 AD 中的复制顺利运行，并确保关键服务正常运行。

2.2.17 Global Catalog(GC)

Global Catalog - Win32 apps | Microsoft Learn

Global Catalog是一个域控制器，用于在 Active Directory 林中存储所有对象的副本。GC 存储当前域中所有对象的完整副本和属于林中其他域的对象的部分副本。标准域控制器保存属于其域的对象的完整副本，但不保存林中不同域的对象的完整副本。GC 允许用户和应用程序查找有关林中任何域中的任何对象的信息。GC 是在域控制器上启用的功能，可执行以下功能：

  • 身份验证（为用户帐户所属的所有组提供授权，在生成访问令牌时包括该授权）
  • 对象搜索（使林中的目录结构透明，允许通过仅提供有关对象的一个属性来跨林中的所有域执行搜索）

2.2.18 Read-Only Domain Controller (RODC)

Read-Only DCs and the Active Directory Schema - Win32 apps | Microsoft Learn

RODC具有只读 Active Directory 数据库。RODC 上不会缓存任何 AD 帐户密码（RODC 计算机帐户和 RODC KRBTGT 密码除外）。不会通过 RODC 的 AD 数据库、SYSVOL 或 DNS 推送任何更改。RODC 还包括只读 DNS 服务器，允许管理员角色分离，减少环境中的replication（复制）流量，并防止将 SYSVOL 修改复制到其他 DC。
